# [#54] Fix HTML dans MDX non rendu Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Permettre l'affichage du HTML inséré manuellement dans les fichiers MDX, actuellement ignoré ou affiché comme texte brut par `TinaMarkdown`.

**Architecture:** TinaCMS stocke le contenu MDX sous forme d'AST rich-text. Les nœuds HTML bruts (`<br>`, `<sup>`, etc.) sont stockés comme nœuds de type `html` avec une propriété `value`. `TinaMarkdown` ne rend pas ces nœuds par défaut — il faut enregistrer un composant `html` dans `getMdxComponents()`. Ce composant utilise `dangerouslySetInnerHTML` (XSS acceptable ici car le contenu vient des éditeurs du référentiel, pas d'utilisateurs externes).

**Tech Stack:** Next.js 14, TinaCMS `TinaMarkdown`, TypeScript

### Task 1 : Ajouter le composant `html` dans `getMdxComponents`

**Fichiers:**

- Modifier: `components/mdx/mdx-components.tsx`

- [ ] **Étape 1 : Vérifier le type de nœud HTML produit par TinaCMS**

  Ouvrir un fichier MDX contenant du HTML manuel (ex. dans `src/content/fiches/fr/`) et lancer le dev server pour observer ce que TinaCMS produit dans l'AST :

  ```bash
  pnpm dev
  ```

  Dans la console navigateur, sur une fiche avec du HTML, inspecter `data.fiches.body` pour confirmer que les nœuds HTML ont `type: "html"` et `value: "<br/>"` (ou similaire).

  Si le type est différent (ex. `html_block`, `html_inline`), adapter l'étape suivante en conséquence.

- [ ] **Étape 2 : Ajouter le rendu HTML dans `getMdxComponents`**

  Dans `components/mdx/mdx-components.tsx`, ajouter dans le retour de `getMdxComponents` :

  ```tsx
  export const getMdxComponents = (lang: Lang = 'fr') => ({
    CTAWithIcon: CTAWithIcon,
    h1: HeadingsH1,
    h2: HeadingsH2,
    h3: HeadingsH3,
    h4: HeadingsStrong,
    h5: HeadingsStrong,
    h6: HeadingsStrong,
    PositionableImage: PositionableImage,
    code_block: CodeBlock,
    // Rendu du HTML brut inséré manuellement dans le contenu MDX
    html: ({ value }: { value: string }) => (
      <span dangerouslySetInnerHTML= />
    ),
    // ... reste inchangé
  ```

- [ ] **Étape 3 : Vérifier les types TypeScript**

  ```bash
  pnpm check-types
  ```

  Résultat attendu : aucune erreur. Si TypeScript se plaint du type `{ value: string }`, utiliser `(props: any)` en attendant une meilleure définition de types TinaCMS.

- [ ] **Étape 4 : Vérifier visuellement**

  ```bash
  pnpm dev
  ```

  Naviguer sur une fiche contenant du HTML manuel. Vérifier que le HTML est bien rendu (ex. `<br>` produit un saut de ligne, `<sup>1</sup>` produit un exposant).

- [ ] **Étape 5 : Vérifier le lint**

  ```bash

  ```

  Résultat attendu : aucune erreur.

- [ ] **Étape 6 : Commit**

  ```bash

  git commit -m "fix(mdx): ajoute le rendu des nœuds HTML bruts dans TinaMarkdown"
  ```

## Note sur la sécurité

`dangerouslySetInnerHTML` est utilisé ici volontairement. Le contenu MDX est géré par les contributeurs du référentiel via TinaCMS, pas par des utilisateurs anonymes. Le risque XSS est acceptable dans ce contexte. Si ce comportement change à l'avenir (contenu utilisateur), ce composant devra être revu.
